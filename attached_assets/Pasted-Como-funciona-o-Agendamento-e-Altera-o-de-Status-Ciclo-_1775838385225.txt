Como funciona o Agendamento e Alteração de Status
Ciclo de vida de um agendamento

O sistema possui 7 status possíveis, e cada transição pode disparar ações financeiras automáticas:

Criado → agendado → confirmado → compareceu → concluido
                              ↘ faltou
              ↘ cancelado

Status	Cor	Descrição	Efeito financeiro
agendado	Azul	Estado inicial de toda consulta	Nenhum
confirmado	Verde	Pré-confirmação administrativa	Nenhum
compareceu	Teal	Paciente chegou	Consome crédito OU gera recebível pendente
concluido	Cinza	Consulta finalizada	Igual ao compareceu (se não disparou antes)
cancelado	Vermelho	Cancelado	Gera crédito de sessão (se plano mensal)
faltou	Laranja	No-show	Taxa de ausência (se habilitado pela clínica)
remarcado	—	Apenas no enum Zod, sem config visual	Nenhum
Onde o status muda

Manualmente (frontend): O usuário clica nos botões do modal de consulta. Cada botão chama handleStatusChange(novoStatus) → PUT /api/appointments/:id → backend executa applyBillingRules().

Botão especial "Concluir": Usa POST /api/appointments/:id/complete — endpoint dedicado que força status = concluido e sempre dispara cobrança.

Automaticamente (CRON — a cada hora): O serviço policyService roda 2 regras:

    Auto-confirmação: Se a clínica configurou autoConfirmHours, consultas no status agendado que estão dentro desse prazo são automaticamente movidas para confirmado
    No-show: Qualquer agendado ou confirmado cujo horário já passou é marcado automaticamente como faltou + gera taxa se noShowFeeEnabled = true na clínica

Lógica de cobrança ao mudar status

A função applyBillingRules() roda toda vez que o status muda e segue esta lógica:

Se novo status = compareceu ou concluido (e não estava assim antes):
  Se billingType = "porSessao":
    Tem crédito disponível?
      SIM → Decrementa 1 crédito + lança R$ 0 (uso de crédito)
      NÃO → Cria recebível PENDENTE com o preço da consulta (ou override da clínica)
Se novo status = cancelado (e não estava assim antes):
  Se billingType = "mensal":
    Gera 1 crédito de sessão para uso futuro + lança R$ 0 (crédito de sessão)

O preço efetivo segue a precedência: override da clínica (procedureCostsTable) > preço base do procedimento.
Validação de conflito ao criar/editar

Dois modelos de conflito:

    Sessão individual (maxCapacity = 1): Qualquer sobreposição de horário no mesmo schedule/clínica bloqueia. cancelado e faltou não contam.
    Sessão em grupo (maxCapacity > 1): Permite N pacientes no mesmo slot, desde que todos compartilhem exatamente o mesmo startTime. Valida capacidade máxima e sobreposição com outros grupos diferentes.

Possíveis melhorias identificadas

    Status remarcado está no enum Zod mas não tem configuração visual (STATUS_CONFIG) — aparece sem badge/cor na UI
    Duplo disparo de cobrança: se um usuário marcar compareceu e depois concluido, a applyBillingRules só cria recebível uma vez (tem proteção contra duplicata), mas vale monitorar
    Auto-confirmação sem billing: confirmado não gera cobrança — correto — mas o usuário pode ter dúvida se "Confirmado" significa que o paciente foi atendido
    No-show via CRON roda a cada hora: um paciente que faltou às 8h terá o status mudado apenas na próxima hora cheia (9h)
